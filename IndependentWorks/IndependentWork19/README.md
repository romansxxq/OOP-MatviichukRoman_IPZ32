# Самостійна робота No19
## Тема: Фабрики + Singleton: заміна без змін клієнта.
## Мета: Навчитися застосовувати патерни Factory Method та Singleton для створення гнучкої системи, де клієнтський код залежить від абстракцій, а не від конкретних реалізацій, що дозволяє легко замінювати компоненти без зміни клієнта.

## Завдання:
Реалізувати систему конвертації файлів (варіант 8), використовуючи комбінацію патернів Factory Method та Singleton. Система повинна підтримувати різні типи конвертації (DocToPdf, JpgToPng, CsvToJson) та дозволяти змінювати активний конвертер під час виконання програми без втручання в логіку клієнта.

### Реалізована структура проєкту:

- IFileConverter — загальний інтерфейс для всіх типів конвертації.

- DocToPdfConverter, JpgToPngConverter, CsvToJsonConverter — конкретні реалізації продукту.

- FileConverterFactory — абстрактний клас фабрики з фабричним методом CreateConverter().

- DocToPdfFactory, JpgToPngFactory, CsvToJsonFactory — конкретні фабрики для створення відповідних конвертерів.

- ConverterManager — клас-singleton, який керує процесом та зберігає посилання на поточну фабрику.

Використання патерну Factory Method дозволило ізолювати процес створення об'єктів-конвертерів. Клієнтський код (клас Program) не викликає оператор new для створення конкретних обробників файлів, а делегує це відповідним фабрикам.

Патерн Singleton забезпечив єдину точку доступу до системи конвертації в межах всього додатку. Поєднання цих патернів реалізує принцип інверсії залежностей (DIP): модулі високого рівня (ConverterManager) не залежать від модулів низького рівня (конкретні конвертери), оскільки обидва залежать від абстракцій (FileConverterFactory та IFileConverter).

Це дає можливість додавати нові формати конвертації (наприклад, Mp3ToWav) просто шляхом створення нових класів, не змінюючи вже існуючий код менеджера чи основної логіки.

### Контрольні питання:

1. Поясніть патерн Factory Method. Які проблеми він вирішує?

Factory Method вирішує проблему жорсткої залежності клієнта від конкретних класів. Він дозволяє системі залишатися незалежною від способу створення, композиції та представлення її об'єктів.

2. Поясніть патерн Singleton. Коли його доцільно використовувати, а коли варто уникати?

Singleton доцільно використовувати для керування спільними ресурсами (логінг, конфігурація, менеджери процесів). Варто уникати, коли він стає "смітником" для глобальних змінних, що ускладнює тестування.

3. Як ці два патерни можуть працювати разом для створення гнучкої та контрольованої
системи?

Разом вони працюють як керована система: Singleton надає глобальний доступ, а Factory Method забезпечує гнучкість наповнення цього доступу різними реалізаціями.

4. Які переваги дає використання абстракцій (інтерфейсів) у поєднанні з Factory Method та
Singleton?

Абстракції дозволяють змінювати поведінку програми під час виконання (Runtime), забезпечують слабку зв'язність (low coupling) та спрощують підтримку коду згідно з принципами SOLID.

### Висновок:
Під час виконання роботи було реалізовано гнучку архітектуру системи конвертації файлів. Завдяки патернам Factory Method та Singleton було досягнуто повної незалежності клієнтського коду від конкретних реалізацій конвертерів. Це дозволяє легко масштабувати систему та замінювати її компоненти без ризику зламати існуючий функціонал.
