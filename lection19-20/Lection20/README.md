# Домашнє завдання (Лекція №20)
## Тема: Adapter, Facade, Proxy: інтеграція й спрощення.

---
### Composite

Патерн Composite дозволяє об’єднувати об’єкти у деревоподібні структури, де кожен елемент може бути як простим (листом), так і складеним (гілкою). Завдяки цьому, ієрархічні структури можна обробляти як єдину колекцію: клієнтський код працює з усіма елементами однаково, не розрізняючи листи та гілки.

**Реальні сценарії застосування:**
1. **Файлова система:** папки містять файли та інші папки, і всі вони мають спільний інтерфейс для операцій (наприклад, “відкрити”, “видалити”).
2. **Графічні інтерфейси:** складні компоненти (панелі, контейнери) містять прості елементи (кнопки, поля), і всі вони обробляються як єдина структура.
3. **Організаційна структура компанії:** менеджери можуть мати підлеглих, які теж можуть бути менеджерами або працівниками, і всі мають спільні методи (наприклад, “отримати зарплату”).

**Переваги:**
- Спрощує роботу з ієрархіями: клієнтський код не залежить від типу елементів.
- Легко розширювати структуру: додавати нові типи компонентів.
- Забезпечує гнучкість та масштабованість.

**Недоліки:**
- Може ускладнити дизайн через надмірну абстракцію.
- Важко контролювати поведінку окремих компонентів, якщо потрібно специфічне оброблення.

Composite — ефективний патерн для роботи з деревоподібними структурами, коли важливо обробляти їх як єдину колекцію.

---
### Adapter, Facade, Proxy: коли застосовувати

**Adapter** використовується для інтеграції несумісних інтерфейсів.

**Реальні сценарії:**
1. Підключення старого API до нового додатку (старий клас адаптується до сучасного інтерфейсу).
2. Використання сторонніх бібліотек, які мають інший формат даних.
3. Інтеграція різних форматів файлів у програмі (наприклад, адаптер для читання XML та JSON).

**Facade** застосовується для спрощення складних підсистем.

**Реальні сценарії:**
1. Старт і налаштування складної системи (наприклад, запуск ігрового рушія через один метод).
2. Робота з багатьма сервісами через єдиний інтерфейс (наприклад, фасад для роботи з базою даних, логуванням і мережею).
3. Спрощення API для користувача, приховуючи внутрішню логіку.

**Proxy** використовується для контролю доступу, кешування або віддаленого доступу.

**Реальні сценарії:**
1. Захист ресурсів (наприклад, proxy для перевірки прав доступу до файлів).
2. Лінивий завантажувач (proxy завантажує об’єкт лише при потребі).
3. Віддалений доступ (remote proxy для роботи з об’єктами на іншому сервері).

**Як патерни вирішують проблеми:**
- Adapter — інтеграція несумісних компонентів.
- Facade — спрощення складних систем для користувача.
- Proxy — контроль доступу, оптимізація, безпека.

**Ключова відмінність:**
- Adapter змінює інтерфейс для сумісності.
- Facade спрощує інтерфейс для зручності.
- Proxy контролює доступ або додає додаткову поведінку при зверненні до об’єкта.

Ці патерни допомагають робити код гнучким, масштабованим і безпечним, вирішуючи різні задачі інтеграції, спрощення та контролю.
